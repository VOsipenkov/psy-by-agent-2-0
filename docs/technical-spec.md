# Техническое задание: Dream Journal 1.0

Актуализировано по фактическому состоянию проекта на март 2026.

## 1. Назначение продукта

Dream Journal - веб-приложение для ведения дневника снов и их интерпретации в формате диалога с AI-ассистентом. Пользователь сохраняет свои сны, отвечает на уточняющие вопросы и получает итоговую трактовку с ключевыми образами и коротким названием сна.

## 2. Цель релиза 1.0

Зафиксировать рабочую первую версию системы, в которой пользователь может:

- зарегистрироваться и войти по логину и паролю;
- вести персональный список снов;
- создавать новый сон и общаться по нему в чат-формате;
- получать уточняющие вопросы или готовую интерпретацию;
- открывать ранее сохраненные сны;
- удалять сон вместе с историей сообщений;
- использовать приложение даже без доступного backend в демо-режиме frontend.

## 3. Границы релиза

### Входит в релиз 1.0

- monorepo с backend, frontend и docker-compose;
- backend на Spring Boot 3.3 + JPA + Flyway + PostgreSQL;
- frontend на React 18 + Vite;
- регистрация и вход по логину/паролю;
- хранение пользователей, снов, сообщений и ключевых слов;
- чат по отдельному сну;
- интеграция с локальной Ollama по HTTP API;
- RAG-контекст для интерпретации снов из локальной базы знаний;
- локальный fallback-анализ, если Ollama недоступна или отвечает некорректно;
- frontend mock mode при недоступности backend;
- seed-данные для демо-входа `admin / admin`.

### Не входит в релиз 1.0

- JWT, refresh token, серверные сессии и полноценная защита API;
- роли пользователей и административная панель;
- потоковая генерация ответа;
- редактирование уже отправленных сообщений;
- поиск по снам, фильтры, теги и архивирование;
- уведомления, e-mail, push и мобильное приложение;
- тестовое покрытие backend/frontend.

## 4. Пользовательские роли

### Пользователь

- регистрируется и входит в приложение;
- работает только со своими снами на уровне UI и маршрутов;
- создает, читает, продолжает и удаляет диалоги по снам.

### Демонстрационный пользователь

- создается автоматически при старте backend;
- логин: `admin`;
- пароль: `admin`;
- содержит 2 предзаполненных примера интерпретированных снов.

## 5. Основные пользовательские сценарии

### 5.1 Регистрация

1. Пользователь открывает экран авторизации.
2. Переключается в режим регистрации.
3. Вводит логин, пароль и подтверждение пароля.
4. Frontend вызывает `POST /api/auth/register`.
5. После успешной регистрации данные пользователя сохраняются в `localStorage`.
6. UI загружает список снов пользователя.

### 5.2 Вход

1. Пользователь вводит логин и пароль.
2. Frontend вызывает `POST /api/auth/login`.
3. Backend проверяет наличие пользователя и корректность пароля.
4. После успешного входа frontend сохраняет пользователя локально.

### 5.3 Первая работа после входа

1. Frontend запрашивает список снов пользователя.
2. Если список пустой, frontend автоматически создает новый сон.
3. Новый сон получает стартовое приветственное сообщение ассистента.

### 5.4 Работа со сном

1. Пользователь выбирает сон из списка или создает новый.
2. Отправляет сообщение с описанием сна.
3. Backend сохраняет сообщение и передает контекст в AI-сервис.
4. AI возвращает либо уточняющий вопрос, либо итоговую интерпретацию.
5. Ответ ассистента сохраняется в истории сна.
6. При финальной интерпретации сохраняются название, ключевые слова и итоговый текст.

### 5.5 Удаление сна

1. Пользователь нажимает кнопку удаления у сна в sidebar.
2. Frontend показывает модальное подтверждение.
3. После подтверждения вызывается `DELETE /api/users/{userId}/dreams/{dreamId}`.
4. Сон и все связанные сообщения/ключевые слова удаляются каскадно.

### 5.6 Работа без backend

1. Если frontend не может достучаться до backend или backend отвечает 5xx, включается mock mode.
2. В mock mode данные хранятся в `localStorage`.
3. Пользователь может продолжать демо-работу без Docker и без backend.

## 6. Функциональные требования

### 6.1 Авторизация

- Логин обязателен, длина от 3 до 40 символов.
- Пароль обязателен, длина от 3 до 100 символов.
- Регистрация запрещает повторное использование логина.
- Вход выполняется по существующему логину и корректному паролю.
- Для старых пользователей без пароля backend умеет один раз сохранить хеш при первом успешном входе.
- Текущее состояние пользователя хранится на frontend в `localStorage`.
- Logout очищает локальные данные без серверного вызова.

### 6.2 Управление снами

- Список снов возвращается по пользователю и сортируется по `updatedAt DESC`.
- Новый сон создается со статусом `NEW`, названием `Новый сон` и стартовым сообщением ассистента.
- Каждый сон можно открыть по `id`.
- Сон можно удалить вместе со всеми дочерними сущностями.
- Если после удаления активного сна список пуст, frontend автоматически создает новый сон.

### 6.3 Чат и жизненный цикл сна

- Сообщение пользователя не может быть пустым и не должно превышать 4000 символов.
- После первого пользовательского сообщения статус сна меняется с `NEW` на `CLARIFYING`.
- AI-ответ сохраняется как сообщение `ASSISTANT`.
- Возможные статусы сна:
  - `NEW`
  - `CLARIFYING`
  - `INTERPRETED`
- После получения финальной интерпретации сон переводится в `INTERPRETED`.

### 6.4 AI-интерпретация

- AI-сервис обращается к Ollama через `/api/generate`.
- В prompt передается история сообщений и RAG-контекст из локальной базы знаний.
- Ответ Ollama должен быть валидным JSON c полями:
  - `stage`
  - `assistantMessage`
  - `title`
  - `keywords`
  - `interpretation`
- Если данных недостаточно, AI должен задать ровно один уточняющий вопрос.
- Если данных достаточно, AI должен вернуть:
  - короткое название сна из 1-2 слов;
  - 2-3 ключевых слова;
  - интерпретацию на русском языке;
  - аккуратную, недогматичную формулировку без медицинских диагнозов.
- Из итогового текста должны удаляться прямые упоминания сонников, авторов и теоретических школ.

### 6.5 Fallback-поведение AI

- Если Ollama недоступна, вернула пустой ответ или ошибку, backend выполняет локальный fallback.
- Если после нескольких сообщений Ollama продолжает возвращать уточнения, backend может завершить анализ локально.
- Fallback должен:
  - выделить ключевые слова из текста пользователя;
  - подобрать разумное название сна;
  - сгенерировать базовую интерпретацию.

### 6.6 Frontend mock mode

- При сетевой ошибке или ответе 5xx frontend переключается в mock mode.
- Mock mode должен поддерживать:
  - регистрацию;
  - вход;
  - список снов;
  - создание сна;
  - отправку сообщений;
  - удаление сна.
- В mock mode используется локальная имитация AI-диалога и seed-данные.

## 7. Нефункциональные требования

- UI должен корректно работать на desktop и mobile.
- Backend и frontend должны запускаться независимо.
- Основной формат API: REST + JSON.
- БД: PostgreSQL.
- Миграции БД: Flyway.
- Запуск полной инфраструктуры должен поддерживаться через Docker Compose.
- Backend должен обрабатывать ошибки в едином JSON-формате.
- CORS должен разрешать локальные адреса frontend для dev-режима.
- Кодовая база должна оставаться готовой к последующему внедрению полноценной безопасности.

## 8. Архитектура решения

### 8.1 Backend

- `auth` - регистрация, вход, bootstrap демо-пользователя;
- `dream` - доменная модель сна, сообщения, CRUD-операции;
- `ai` - сбор prompt, RAG, интеграция с Ollama, fallback-анализ;
- `common` - базовые сущности, исключения, обработка ошибок;
- `config` - парольный encoder, CORS, HTTP-клиент, свойства Ollama.

### 8.2 Frontend

- единое SPA-приложение;
- экран авторизации с режимами login/register;
- основной workspace с sidebar, чатом и панелью интерпретации;
- API-слой для работы с backend;
- fallback API-слой в mock mode;
- хранение текущего пользователя и mock-данных в `localStorage`.

### 8.3 Инфраструктура

- `postgres` - основная БД;
- `ollama` - локальный AI runtime;
- `backend` - Spring Boot API;
- `frontend` - React/Vite + nginx container.

## 9. Модель данных

### `users`

- `id: uuid`
- `username: varchar(40), unique, not null`
- `password_hash: varchar(100), nullable для совместимости`
- `created_at`
- `updated_at`

### `dream_conversations`

- `id: uuid`
- `user_id: uuid -> users.id`
- `title: varchar(120), not null`
- `stage: varchar(32), not null`
- `interpretation: text`
- `created_at`
- `updated_at`

### `dream_messages`

- `id: uuid`
- `conversation_id: uuid -> dream_conversations.id`
- `role: USER | ASSISTANT | SYSTEM`
- `content: text, not null`
- `created_at`
- `updated_at`

### `dream_keywords`

- `conversation_id`
- `position_index`
- `keyword`

## 10. REST API

### Авторизация

- `POST /api/auth/login`
- `POST /api/auth/register`

Тело запроса:

```json
{
  "username": "roberto",
  "password": "secret123"
}
```

Ответ:

```json
{
  "id": "uuid",
  "username": "roberto"
}
```

### Сны

- `GET /api/users/{userId}/dreams`
- `POST /api/users/{userId}/dreams`
- `GET /api/users/{userId}/dreams/{dreamId}`
- `POST /api/users/{userId}/dreams/{dreamId}/messages`
- `DELETE /api/users/{userId}/dreams/{dreamId}`

Тело отправки сообщения:

```json
{
  "content": "Мне снилось, что я бегу по темному лесу и ищу дом."
}
```

## 11. Формат ошибок API

При ошибках backend возвращает JSON вида:

```json
{
  "timestamp": "2026-03-25T12:00:00Z",
  "status": 400,
  "error": "Bad Request",
  "message": "Неверный логин или пароль"
}
```

## 13. Критерии приемки релиза 1.0

- Пользователь может зарегистрироваться и войти.
- После входа открывается список снов пользователя.
- При отсутствии снов автоматически создается новый.
- Пользователь может отправить сообщение и получить ответ ассистента.
- После накопления контекста по сну сохраняются `title`, `keywords`, `interpretation`.
- Пользователь может открыть ранее созданный сон.
- Пользователь может удалить сон с подтверждением.
- При недоступности backend frontend остается работоспособным в mock mode.
- Сборка frontend проходит командой `npm run build`.
- Backend успешно проходит `mvn test` как минимум на уровне компиляции и запуска build lifecycle.

## 14. Известные ограничения текущей версии

- Сервер пока не использует JWT и не защищает API токенами.
- Идентификация пользователя строится на `userId` в URL и локальном хранении пользователя на frontend.
- Нет автоматических backend-тестов и frontend-тестов.
- Нет истории версий сна, редактирования и поиска.
- Нет streaming-ответов от AI.
- Качество интерпретации зависит от локальной модели Ollama и качества промпта.
